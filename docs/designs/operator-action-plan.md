# Operator Action Plan — Design

**Status:** Draft
**Author:** architect
**Date:** 2026-04-05
**Audience:** Antellion operator (Jordan), future engineers building on report infrastructure

---

## 1. Problem

Antellion sells $10K–$15K executive audit reports. The report is the deliverable — but it is not the consulting engagement. Between "report generated" and "client meeting," the operator needs to:

1. **Validate** findings that could embarrass the firm if wrong.
2. **Rehearse** the narrative — which 3–5 findings lead the conversation.
3. **Pre-empt pushback** — know what the VP TA buyer will challenge and how to respond.
4. **Spot upsell hooks** — where does this report imply a follow-on engagement.
5. **Flag weak claims** so they can be downplayed or omitted verbally.

Today this lives in Jordan's head. That is fine at n=1, but:

- It does not survive a second operator (future hire, contractor).
- It does not survive a two-week gap between report generation and delivery.
- It wastes the rich metadata already sitting in `Report.metadata` — readiness warnings, stability distribution, confidence scores, journey analysis, citation gaps are all computed and stored, then only surfaced through the client-facing renderer.

## 2. Artifact Definition

**The Operator Action Plan is a private, internal-only briefing rendered from existing `Report.metadata`, shown as an additional tab on the report detail page.**

### Explicit non-goals

- It is **not** a second report. No cover page, no export pipeline, no client-safe phrasing.
- It is **not** a new document stored separately. Zero new DB tables, zero schema changes.
- It is **not** a place for the operator to write freeform notes — that is a separate concern (and if needed later, a `Report.operatorNotes` text field is one migration away).
- It is **not** generated by a new LLM call in Phase 1. Everything in MVP is deterministic, derived from metadata the composer already writes.

### What it is

A new route `apps/web/src/app/(dashboard)/reports/[id]/action-plan/page.tsx` that:

- Reads `Report.metadata` + `Report.recommendations` + the client/competitor context.
- Runs a pure transform function from `packages/core` that yields an `OperatorActionPlan` object.
- Renders seven sections with unmistakable "INTERNAL — NOT CLIENT-VISIBLE" chrome (red border top, red badge in header, `robots: noindex`, distinct background color).
- Is printable to a single standalone briefing (roughly 2–4 pages) via a `/print` variant of the same route.
- Is accessible only to the operator's organization (same org scoping as the report — multi-org ready).

### Why a tab, not a separate route tree

- Zero duplication of auth, fetching, `notFound()`, org scoping.
- Same URL shape as the existing `/reports/[id]/qa` sibling — the pattern already exists.
- Operator muscle memory: from any report, one click away.

## 3. Data Sources

The hard rule for Phase 1: **if it is not already in `Report.metadata`, we do not surface it.** Every section below maps to existing data written by `generateReport()` in `apps/web/src/app/(dashboard)/actions/reports.ts`.

| Section                         | Source in `Report.metadata`                                                                                     | Notes                                                                                                                                                         |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Findings to Validate            | `readinessWarnings[]`, `confidence.perSection`, `multiRunAnalysis.stabilityDistribution`, `overallSourcedRate`  | Already written at lines 1265–1305 of `actions/reports.ts`. The action plan re-ranks and surfaces them as "go verify these manually before sending."          |
| Talking Points                  | `journeyAnalysis.stages[]` (mentionRate, positioning, narrative, competitorCallout), `journeyAnalysis.criticalGapStage`, `overallThemes`, top recommendations | Deterministic ranking: pick the 3–5 stages/competitors/themes with the sharpest, highest-confidence numbers. The narrative text is already written by `journey-metadata-builder.ts`. |
| Predicted Pushback              | Pattern-matched against `journeyAnalysis` + `competitors[].threatLevel` + `contentAssetTypes`                   | Phase 1: static rule-based mapping (finding type → likely pushback → canned response using the specific numbers from this report). Phase 2: LLM synthesis.   |
| Upsell Opportunities            | `remediationPlan.criticalCount`/`highCount`, `stabilityDistribution.VOLATILE_PRESENCE`, `multiRunAnalysis.effectiveScanRunCount`, existing `ContentAsset` count | Deterministic rules: many criticals → content engagement; volatile presence → quarterly monitoring; single scan → re-scan upsell; snapshot report → full assessment upgrade. |
| Red Flags                       | `readinessWarnings[]` (severity=critical), `confidence.overall.tier === "LOW"`, sections with fewer than N supporting results, citation rate < 0.3, stages with zero results | Every one of these is already computed. This section is a re-presentation, not new logic.                                                                    |
| Next Engagement                 | Derived from criticalGapStage + topPriorityStage + report type (snapshot vs full)                               | Static playbook of "if criticalGapStage === DISCOVERY, recommend X follow-up; if criticalGapStage === EVALUATION, recommend Y."                               |
| Questions to Ask the Client     | Same trigger flags that feed Upsell + Next Engagement (no additional metadata)                                  | Rule-based trigger → question template map. Shares `TriggerFlags` with upsell rules so there is exactly one place where "which conditions fired" is computed. |

**Net new LLM calls in Phase 1: zero.** The artifact is purely a new view over existing data.

## 4. Content Structure

Each section is designed to answer one specific question the operator has in the 24 hours before a client meeting. Order matters — it mirrors the actual prep workflow.

### A. Findings to Validate Manually (red, top of page)

A checklist of specific claims in the report that a human should eyeball before sending. Each item has a **target** (the claim in the report), a **why** (why it might be wrong), and a **verification step** (exactly what to check).

**Inclusion rules (all deterministic):**
- Any section whose `confidence.perSection[key].tier === "LOW"`.
- Any journey stage whose `sourcedRate < 0.3` — the claim depends on AI parametric memory, not indexed sources.
- Any competitor callout where `topCompetitor.mentionRate - clientMentionRate > 0.2` AND the competitor's gap platform list contains unfamiliar domains (sanity check the competitor actually has that presence).
- Any stage where `resultCount < 5` — insufficient sample to make a public claim.
- Any readiness warning of severity `critical` or `warning`.
- Any finding where `multiRunAnalysis` shows the query as `VOLATILE_PRESENCE` — it flipped between runs, so a third run might show the opposite.

Each item renders as:

```
[ ] VERIFY — Plaid Discovery visibility = 12%
    Why suspect: Based on 7 Discovery queries (sample size below threshold).
                 Sourced rate = 22% — AI was largely recalling from memory.
    How to check: Re-run the 3 highest-signal Discovery queries manually in
                  ChatGPT + Claude web. Confirm Plaid is indeed absent.
                  Spot-check competitor Stripe's Glassdoor page exists as claimed.
```

### B. Talking Points (lead-with narrative)

The 3–5 findings the operator should lead with in the client meeting, ranked by a deterministic "compelling score". Every factor is normalized to `[0, 1]` so each contributes equally to the product:

```typescript
// Normalized to [0, 1] so each factor contributes equally
const normalizedSample = Math.min(Math.log(Math.max(sampleSize, 1)) / Math.log(50), 1);

compellingScore =
  confidence *           // 0..1
  normalizedSample *     // 0..1 (log-scaled, caps at sampleSize=50)
  gapMagnitude *         // 0..1 (|clientRate - topCompetitorRate|, already bounded)
  sourcedRate;           // 0..1
```

**Why the cap is 50:** sample sizes beyond 50 do not meaningfully change credibility for a VP TA buyer. At 50 queries, a finding is already defensible in an executive meeting. Beyond that, additional data is diminishing returns and should not dominate the ranking — otherwise a high-sample weak finding would out-rank a lower-sample sharp finding, which is the opposite of what the operator needs.

For each talking point:
- **The number to memorize** (e.g., "12% Discovery visibility vs. Stripe's 67%")
- **The exact narrative sentence** (already generated by `journey-metadata-builder.ts` — reuse `stage.narrative` verbatim)
- **A quote lever** — if `overallThemes` has negative attributes, pull the sharpest one
- **The anchor recommendation** — which of the existing `report.recommendations` pairs with this finding

No LLM in Phase 1. The narrative text is already written; the action plan just curates which lines to rehearse.

### C. Predicted Pushback (the pre-mortem)

A list of probable objections with pre-written responses. Phase 1 uses a static rule table that pattern-matches on the report shape:

| Trigger condition                                                 | Predicted pushback                                             | Scripted response (fills in specific numbers)                                                                       |
| ----------------------------------------------------------------- | -------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------- |
| `contentAssetTypes` includes "Glassdoor" AND Discovery gap exists | "But we already invest heavily in Glassdoor"                   | "Right — and that's exactly why the gap is surprising. In {n} Discovery queries, AI cited glassdoor.com but not {clientName}. The presence exists; the authority signal does not." |
| Single `scanCount`                                                | "How do I know this isn't just one bad day with the AI?"      | "Fair question. This report is built from {n} queries across {m} models. For any finding in the report, we can re-run it live. We also track stability — {k}% of queries in this set were validated across multiple runs." |
| `criticalGapStage === "EVALUATION"`                              | "Our recruiters say candidates love us once they're talking"  | "That's consistent with what we found — it's the stages *before* the recruiter call where the pipeline is leaking. {funnelThroughput}% of AI-researching candidates reach the comparison stage with you in the running." |
| High competitor threat level                                     | "We're bigger than competitor X, why are they beating us here?" | "Size doesn't correlate with AI surfacing. Competitor X has {specific platform presence}, which is what AI indexes. Revenue and headcount aren't what the model is trained on." |
| `overallSourcedRate < 0.4`                                       | "Where is this data coming from?"                              | "The responses themselves are the data — {n} AI outputs. Sourcing rate is intentionally one of our metrics: a {pct}% sourced rate tells us AI is recalling you from training memory rather than fresh indexed content. That's a finding, not a weakness in the method." |

Phase 2 upgrades this to an LLM synthesis call for edge cases not covered by the static rules.

**Known coverage gaps.** The pushback table only covers objections that can be triggered from report metadata. It intentionally does not cover **situational objections** — the most common real objection in enterprise sales:

- "We're in a hiring freeze."
- "This isn't a priority this quarter."
- "Budget cycle is locked until Q3."
- "We just got a new CHRO — she wants to set her own agenda first."

These are not report-derived. No combination of `journeyAnalysis`, `confidence`, or `competitors` tells us Plaid froze hiring last week. Attempting to trigger them from metadata would produce false positives and undermine trust in the rest of the action plan.

**Where they belong instead:** a separate sales onboarding playbook (not in scope for this design). The action plan handles report-derived objections; the onboarding playbook handles situational ones. Future work: link the two from the operator view so the briefing surfaces both on a single screen. For now, the operator is expected to hold situational responses in their head or in their own notes.

### D. Upsell Opportunities

Deterministic mapping from report shape → follow-on engagement:

| Condition                                                          | Follow-on to propose                               | One-liner to drop in the meeting                                                                             |
| ------------------------------------------------------------------ | -------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| `remediationPlan.criticalCount >= 3` and most are content gaps     | Content authoring retainer ($X/month)              | "We've got the diagnosis. The remediation is executable — we can draft the first three assets for you."     |
| `stabilityDistribution.VOLATILE_PRESENCE / totalQueries > 0.3`     | Quarterly monitoring subscription                  | "Your visibility is unstable. A quarterly re-scan will tell you whether our recommendations are moving it." |
| `scanCount === 1` (snapshot-only)                                 | Full assessment upgrade                            | "This was a snapshot across {n} queries. The full assessment runs {m}+ and includes the remediation playbook." |
| `criticalGapStage` exists AND client has no existing `ContentAsset`s for the corresponding platform | Platform-specific content creation sprint          | "You have nothing on {platform}. That's a 2–3 week build we can own."                                        |
| Multiple stages with CONTENDER/PERIPHERAL positioning              | Strategic advisory retainer                        | "Your issue isn't any single stage — it's that you're not differentiated anywhere. That's a 3-month brand positioning problem, not a content problem." |

Each upsell row shows the trigger condition that fired, so the operator knows *why* this is a live opportunity for this specific client.

### E. Red Flags / Quality Concerns

The operator's "do not get caught" section. All readiness warnings of severity critical, plus any of:

- Confidence tier LOW anywhere in `confidence.perSection`.
- Citation rate < 30% globally.
- Any stage with zero results (report is making a claim with no sample).
- Any narrative that used `hedgePhrase` fallback language.
- Any recommendation whose `evidenceBasis` is empty or generic.

Each red flag shows: the concern, the specific location in the report it affects, and a suggested mitigation.

**Severity classification.** Red flags are not equal. Different flags demand different mitigations, and a flat list hides the critical items beneath advisory noise. Every `RedFlag` carries a `severity: "critical" | "major" | "advisory"` field and the section is rendered sorted critical-first.

| Severity    | Mitigation posture                                                                                    | Classification rule                                                                                                                 |
| ----------- | ----------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| `critical`  | Deleting or heavily caveating the affected section before sending is warranted.                       | Zero sourced results for an entire stage; confidence tier LOW on a stage the report leads with; a quantified claim with `resultCount === 0`. |
| `major`     | Verbal qualification during the walkthrough is warranted ("our Discovery sample on this was thin…"). | Sample below threshold but non-zero; stage-level sourced rate < 0.3; volatile presence > 30% for a lead stage.                      |
| `advisory`  | Prepare an answer but do not volunteer the concern unless the client raises it.                      | Global sourced rate 0.3–0.4; generic evidenceBasis on a non-lead recommendation; single-run report where multi-run would strengthen it. |

The `RedFlag` type in `types.ts` carries the severity explicitly. The render layer sorts `critical → major → advisory` and visually separates critical items (red left border) from the rest (amber / neutral).

### F. Suggested Next Engagement

A short, opinionated "what to propose in the meeting." Chooses one primary recommendation from the upsell list above by walking a named priority constant, `DEFAULT_UPSELL_PRIORITY`, defined in `packages/core/src/operator-action-plan/rules/next-engagement.ts`:

```typescript
// Ordering rationale:
//   1. content        — most immediately billable and actionable; the operator
//                       can scope and start a content sprint inside one week.
//   2. monitoring     — creates recurring revenue and a reason to re-engage
//                       every quarter; second-best because it compounds.
//   3. advisory       — strategic retainer; highest ceiling but hardest to
//                       scope out of a single meeting, so a weaker lead.
//   4. snapshotUpgrade — lightest-touch follow-on; use only when nothing
//                        else fires.
//
// This ordering is a HYPOTHESIS. Revisit after 3–5 real deliveries based on
// which engagements Jordan actually closes. If monitoring closes more than
// content, flip the first two.
export const DEFAULT_UPSELL_PRIORITY = [
  "content",
  "monitoring",
  "advisory",
  "snapshotUpgrade",
] as const;
```

The next-engagement rule walks `DEFAULT_UPSELL_PRIORITY` in order and picks the first upsell whose trigger fired for this report. Renders:

- **Recommended next step:** (one of ~5 templates)
- **Proposed timeline:** 2 weeks / 1 month / ongoing
- **Materials to prep:** what the operator should bring to a second conversation (e.g., "sample Glassdoor response drafts," "competitor deep-dive on Stripe," "quarterly monitoring dashboard mock")
- **Reference price range:** (from a config constant, not hardcoded per report)

### G. Questions to Ask the Client

The plan so far covers what to **say** (talking points) and what to **defend** (pushback, red flags). It does not yet cover what to **probe**. A VP TA report meeting is also a discovery call for the follow-on engagement, and walking in with 2–3 prepared questions turns the walkthrough into structured scoping without the operator having to improvise.

**Purpose.** Turn the report review meeting into a discovery call for the follow-on engagement. The questions are not for the buyer's benefit — they are for the operator to learn what internal ownership, tooling, and appetite already exist so the upsell lands in the right place.

**Generation.** Deterministic. The same trigger conditions that fire in Sections D (Upsell) and F (Next Engagement) also fire question templates. If an upsell trigger fires, the paired question fires. This keeps the two sections aligned — every live upsell has a discovery question attached.

**Trigger → Question map (rule-based, Phase 1):**

| Trigger condition                                                          | Question to ask                                                                      |
| -------------------------------------------------------------------------- | ------------------------------------------------------------------------------------ |
| Glassdoor gap detected (Evaluation stage, Glassdoor absent from assets)    | "Who owns your Glassdoor response strategy today?"                                   |
| No baseline / first assessment the client has run                          | "How are you currently tracking employer brand performance?"                         |
| Critical pipeline stage gap (`criticalGapStage` set)                       | "Which recruiting channels are you relying on most for {stage}?"                     |
| Strong competitor contrast (`topCompetitor` > 20pt gap)                    | "Have you done any structured benchmarking against {competitor} before?"             |
| Content gap (content-type upsell fired)                                    | "Do you have a content creation pipeline for career-focused content today?"          |
| Zero owned citations (`overallSourcedRate < 0.1` for client URLs)         | "Who manages your employer brand content strategy internally?"                       |
| Volatile presence > 30% (monitoring upsell fired)                          | "How often does your team revisit your employer brand positioning?"                  |

**Output.** 2–3 questions per report (hard cap — more than 3 in a briefing dilutes focus). The build function walks the trigger list, collects matching questions, and takes the first 3 in the same priority order as `DEFAULT_UPSELL_PRIORITY` so the questions align with the primary upsell.

**Implementation.** Data table in `packages/core/src/operator-action-plan/rules/questions.ts`, same shape as `pushback.ts` and `upsell.ts`. No new LLM call. The trigger detection logic is shared with the upsell rules — `buildQuestions()` and `buildUpsell()` consume the same `TriggerFlags` object so there is one source of truth for "which conditions fired on this report."

## 5. UI / Access Pattern

### Route

`/reports/[id]/action-plan` — sibling to the existing `/reports/[id]/qa` route.

### Navigation

Add a third tab on the report detail page header:
`Report · QA Review · Action Plan (Internal)`

The "Action Plan (Internal)" tab has a red dot and subdued red text to signal it is not client-visible.

### Visual chrome

Aggressive visual distinction — the goal is that it is *impossible* for the operator to accidentally screenshot this and send it to a client:

- Full-page red border top (8px).
- Fixed banner at top: **"INTERNAL BRIEFING — NOT FOR CLIENT DELIVERY"** in white on red.
- Different background color from the report view (warm off-white vs. white).
- Watermark on print: "INTERNAL — ANTELLION OPERATOR ONLY" diagonally across the page.
- `<meta name="robots" content="noindex, nofollow">`.
- No export-to-PDF button in the client export pipeline — action plan has its own print route with the watermark baked in.

### Print view

`/reports/[id]/action-plan/print` — same transform, print-optimized layout. Jordan wants this in his hand on a flight, not on a laptop screen.

### Access control

Same org scoping as the report (`where: { id, client: { organizationId } }`). In the future when multi-org auth is real, this section should respect a per-user role check — only users with `role: "OPERATOR"` or `role: "OWNER"` see the tab. Design note the check in the code; do not build role infrastructure yet.

**`robots: noindex` is belt-and-suspenders, not security.** The actual access control is the org-scoped Prisma query — the same pattern already enforced by the existing report detail route. The action plan route follows that exact pattern (`where: { id, client: { organizationId } }`) and inherits the same guarantees. The `<meta name="robots" content="noindex, nofollow">` tag only prevents the URL from being crawled by well-behaved search engines if it somehow escapes auth; it does not prevent an unauthorized authenticated user from viewing the page. Future engineers must not treat the meta tag as a security mechanism. If the org scoping query is ever weakened, the noindex tag provides no defense.

## 6. When It Is Generated

**Never as a stored artifact in Phase 1.** The action plan is derived every time the page loads, from `Report.metadata`. This has three important properties:

1. **No regeneration flow** — if the report metadata changes (re-generation, re-scan), the action plan updates automatically.
2. **No stale cache** — no "stale action plan" bug to chase.
3. **Zero migration risk** — we never write operator-specific data into a schema that is also used client-facing.

Phase 2 introduces an optional cache (simple JSON blob in `Report.metadata.operatorActionPlan`) only if the pure transform becomes expensive. As of today, all inputs are already in memory when the page loads — the transform is a few hundred microseconds.

## 7. Implementation Plan

### Phase 1 — MVP (1–2 days, ship before next client delivery)

**Core package (`packages/core/src/operator-action-plan/`):**
- `types.ts` — `OperatorActionPlan`, `ValidationItem`, `TalkingPoint`, `PredictedPushback`, `UpsellOpportunity`, `RedFlag` (with `severity: "critical" | "major" | "advisory"`), `NextEngagement`, `ClientQuestion`, `TriggerFlags`.
- `config.ts` — named constants for price ranges (e.g., `CONTENT_AUTHORING_PRICE_RANGE`), timeline defaults, and the `COMPELLING_SCORE_SAMPLE_CAP = 50`. Rule tables reference these constants rather than embedding the values. Reviewed quarterly.
- `build.ts` — pure function `buildOperatorActionPlan(input: OperatorActionPlanInput): OperatorActionPlan`. Input is a subset of `Report` + client + competitors + `contentAssetTypes`. Computes `TriggerFlags` once and passes it to upsell, next-engagement, and questions builders.
- `rules/pushback.ts` — the static rule table for predicted pushback (data, not code — JSON literal array).
- `rules/upsell.ts` — the static rule table for upsell opportunities. References price constants from `config.ts`.
- `rules/next-engagement.ts` — playbook templates plus the exported `DEFAULT_UPSELL_PRIORITY` constant.
- `rules/questions.ts` — the trigger → client-question table for Section G.
- `index.ts` — public exports.
- `build.test.ts` — snapshot tests covering: strong-report input, weak-report input, snapshot-scan input, journey-format input, legacy-format input. Includes explicit severity-ordering assertion for red flags.

**Web app:**
- `apps/web/src/app/(dashboard)/reports/[id]/action-plan/page.tsx` — server component, fetches the report and calls `buildOperatorActionPlan()`.
- `apps/web/src/app/(dashboard)/reports/[id]/action-plan/print/page.tsx` — same data source, print layout.
- `apps/web/src/components/operator/` — new components: `ActionPlanHeader`, `ValidationChecklist`, `TalkingPointsCard`, `PushbackCard`, `UpsellCard`, `RedFlagsCard` (sorts by severity critical-first), `NextEngagementCard`, `ClientQuestionsCard`.
- Add the third tab to the existing report detail page header.

**Explicit out-of-scope for Phase 1:**
- No LLM calls.
- No operator notes / freeform text.
- No per-user role check (assume single-operator org).
- No database schema changes.
- No export pipeline integration — print works, export does not.

**Config-constants rule (non-negotiable).** Price ranges (e.g., `"$8K-$12K for content authoring"`), timelines, and scoring caps live **only** in `/packages/core/src/operator-action-plan/config.ts` as named constants. Rule tables (`pushback.ts`, `upsell.ts`, `next-engagement.ts`, `questions.ts`) reference the constant by name; they do not duplicate the literal value. This prevents the situation where pricing changes in one table and drifts in another. Constants are reviewed quarterly — schedule a recurring review with the pricing conversation, not the engineering one.

### Phase 2 — LLM synthesis (ship after 3–5 real deliveries)

Criteria for adding LLM: the static pushback table is obviously missing coverage on real client meetings. Without that signal, do not build this.

- Add `packages/prompts/src/operator/pushback-synthesis.ts` — prompt that takes the journey analysis + top recommendations and emits 2–3 additional pushback/response pairs the static table missed.
- Add optional cache field `Report.metadata.operatorActionPlan.syntheticPushback`.
- Add "Regenerate with AI synthesis" button on the action plan page (manual trigger, not automatic).

### Phase 3 — Historical learning (ship after 10+ deliveries, maybe never)

- Add a lightweight feedback loop — operator marks which talking points landed in the client meeting, which pushbacks actually surfaced, which upsells converted.
- Store as `OperatorActionPlanFeedback` table (the first and only new schema).
- Use the history to rank talking points and pushback rules by hit rate, feed back into the build function.

This phase is speculative and should not drive any Phase 1 design decisions.

## 8. Tradeoffs and Risks

**Risk: the static rule tables drift from what real clients say.**
Mitigation: the rule tables are in `packages/core/src/operator-action-plan/rules/*.ts` as plain JSON literals. Editing them is a one-line change. After every real client meeting, add/update one row.

**Risk: operator accidentally shares internal content.**
Mitigation: aggressive visual chrome (red border, watermark, banner). No path from this view into the export pipeline.

**Risk: the deterministic scoring misses the actually-interesting finding.**
Mitigation: the talking points section is a recommendation, not a constraint. The operator can ignore it. Worst case the operator still has the full report.

**Risk: adding a new tab clutters the report detail page for future non-operator users.**
Mitigation: tab is conditionally rendered when the session indicates operator role. Stubbed as `true` in Phase 1; becomes a real check when multi-org auth ships.

**Risk: the content-asset-aware upsell logic duplicates what lives in the recommendation engines.**
Mitigation: import the same `existingAssetUrls` check already used by the journey recommendations. Do not re-implement.

## 9. Example Output — Hypothetical Plaid Scan

*This is what the rendered page would look like for a mid-quality Plaid report.*

---

**[RED BANNER]** INTERNAL BRIEFING — NOT FOR CLIENT DELIVERY

### Plaid · Operator Action Plan
Generated from report · April 5, 2026 · 4 scan runs · Journey format

---

#### A. Findings to Validate Manually (3 items)

- [ ] **VERIFY — Plaid Discovery visibility = 12%**
  Why suspect: 7 Discovery queries (below 10-query threshold). Sourced rate 22% — AI relying on parametric memory.
  How to check: Re-run the three highest-signal Discovery queries manually in ChatGPT web + Claude web. Confirm Plaid is genuinely absent from AI-generated "top fintech employers" lists.

- [ ] **VERIFY — Stripe's Glassdoor dominance claim**
  Why suspect: Report asserts Stripe "has deep Glassdoor presence" driving the Evaluation gap. The claim is stage-appropriate but we are attributing causation.
  How to check: Open glassdoor.com/Overview/Working-at-Stripe. Count review volume. Compare to glassdoor.com/Overview/Working-at-Plaid. Confirm the delta is real before leading with it.

- [ ] **VERIFY — The "compensation detail: absent" theme**
  Why suspect: `overallThemes.compensationDetail === "absent"`. This is a strong verbal talking point but only if true.
  How to check: Ask ChatGPT "What does Plaid pay senior engineers?" directly. If it gives specific numbers, the theme is wrong and must be walked back.

---

#### B. Talking Points (lead with these)

**1. Funnel collapse at Discovery (primary lead)**
  Number: **12% Discovery visibility. 67% for Stripe. 54-point gap.**
  Narrative: "Plaid has partial Discovery visibility at 12%, appearing in some but not most AI-generated employer recommendations. Stripe leads Discovery by 55 points — Stripe has presence on glassdoor.com, levels.fyi, builtin.com, platforms where Plaid has limited or no data."
  Anchor recommendation: "Establish Glassdoor authority via employee review campaign"

**2. Evaluation stage exclusion**
  Number: **Plaid appears in 18% of Evaluation comparison queries.**
  Narrative: "Candidates deciding between employers are making that decision without Plaid in the running."
  Anchor recommendation: "Create head-to-head comparison content for top 3 evaluation competitors"

**3. Missing compensation detail**
  Number: Compensation detail classified as "absent" across 47 response texts.
  Narrative: "When AI describes working at Plaid, it describes culture in vague terms and compensation not at all. Stripe's AI profile includes specific salary bands."
  Anchor recommendation: "Publish verified compensation bands on levels.fyi"

---

#### C. Predicted Pushback (pre-mortem)

**"We already invest heavily in Glassdoor"** (expected — `contentAssetTypes` includes Glassdoor + Discovery gap exists)
  Response: "Right — and that's exactly why the gap is surprising. In 4 of 7 Discovery queries, AI cited glassdoor.com but not Plaid specifically. The presence exists; the authority signal does not. That's a content and review-volume problem, not a 'do we have a page' problem."

**"Our recruiters say candidates love us once they're talking"** (expected — `criticalGapStage === "EVALUATION"`)
  Response: "That's consistent with what we found — it's the stages *before* the recruiter call where the pipeline is leaking. At current rates, 14% of AI-researching candidates reach the comparison stage with Plaid in the running. The issue isn't closing; it's being on the list."

**"We're bigger than Mercury and Ramp, why are they beating us?"** (expected — competitor threat)
  Response: "Size and AI surfacing aren't correlated. Mercury has dense presence on builtin.com and blind.app — two platforms AI indexes heavily for fintech. Plaid's revenue doesn't show up in the training data the way a 500-review Glassdoor page does."

---
