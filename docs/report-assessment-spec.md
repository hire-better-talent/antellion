# AI Employer Visibility Assessment -- Report Specification

**Status:** Actionable specification
**Purpose:** Align the Antellion report output with the paid "AI Employer Visibility Assessment" deliverable defined in the growth playbook
**Audience:** Frontend developers, report PM, growth lead
**Date:** March 2026

---

## 1. CURRENT STATE ANALYSIS

### What the report produces today

The `composeReport()` function in `packages/core/src/report-composer.ts` generates a report with:

| Element | Current content |
|---|---|
| **Title** | "AI Visibility Audit -- [Client] -- [Month Year]" |
| **Executive summary** | 2-4 sentences: mention rate, visibility score, top competitor gap, recommendation count |
| **Section 1: Visibility findings** | Mention count, mention rate, visibility score, sentiment label, qualitative tier (low/moderate/strong) |
| **Section 2: Competitor analysis** | Competitors above/below client mention rate, largest gap in percentage points |
| **Section 3: Citation patterns** | Total domains, gap domains (list), client-exclusive domains (list), shared domains (list) |
| **Recommendations** | 3-6 recommendations with category, priority, title, description, impact, effort |

### What the export page renders

The export page (`apps/web/src/app/reports/[id]/export/page.tsx`) renders:
- A document header with "Antellion" branding, title, client name/domain/industry, date
- Executive summary as a paragraph
- Sections as heading + paragraph blocks
- Recommendations as a table (number, title/description/impact, category, priority, effort)
- Footer with confidentiality line and "Generated by Antellion"

### Structural gaps

The current report reads like a **system output summary** -- it concatenates data points into paragraphs. It does not read like a **strategic deliverable** that a VP of Talent Acquisition would present to their CPO or board. Specific problems:

1. **Title says "Audit" instead of "Assessment"** (playbook explicitly says to avoid "audit")
2. **Executive summary is data-recitation, not strategic framing** -- it lists numbers without interpreting what they mean for the business
3. **Three flat sections with no hierarchy** -- a 3-card grid in the dashboard, sequential paragraphs in export. No narrative arc from "here is the situation" to "here is what it means" to "here is what to do"
4. **No methodology section** -- enterprise buyers need to trust the process before trusting the findings
5. **No company/assessment scope section** -- no record of what was assessed (how many queries, which themes, which competitors, which AI models)
6. **Citation section is a domain list, not analysis** -- lists domains but does not explain what type of source they are or why they matter
7. **Competitor section lacks per-competitor detail** -- aggregates competitors into "above" and "below" client, but does not give each competitor their own profile
8. **No query intent breakdown** -- the 6 query themes (reputation, compensation, hiring process, role expectations, culture, competitor comparison) are not surfaced in the report at all
9. **Recommendations lack "what to do first" framing** -- they are sorted by priority but lack a clear "start here" narrative
10. **No bonuses delivered** -- the playbook promises Competitor Visibility Scorecards, Citation Source Playbook, and Query Intent Map as bonuses, none of which exist in the report
11. **No cover page** -- the export jumps straight into content without a proper title page
12. **Sections are plain text only** -- no tables, no comparison charts, no visual hierarchy within sections

## 2. TARGET REPORT STRUCTURE

The Assessment report should follow this exact hierarchy. Each section is specified with content, data source, buyer outcome, and language guidance.

### COVER PAGE

**Content:**
- "Antellion" wordmark (top)
- Report title: "AI Employer Visibility Assessment"
- Client name and logo (if available)
- Industry
- Assessment date range (scan dates)
- "Confidential -- Prepared for [Client Name]"
- "Prepared by Antellion"

**Data source:** `client.name`, `client.domain`, `client.industry`, `client.logoUrl`, `report.generatedAt`

**Buyer outcome:** Sets the tone. The buyer opens this and immediately sees a branded, professional document that signals consulting-grade quality. This is the page they see when they share the PDF with their leadership team.

**Language guidance:**
- No subtitle. The category name does the work.
- Do NOT include version numbers, report IDs, or internal metadata
- Do NOT say "Draft" or "Review" -- the exported version should always feel final

---

### 1. EXECUTIVE SUMMARY (1 page)

**Content:**
- Opening statement: what was assessed and why it matters (2-3 sentences)
- Key findings (3-5 bullets, each one sentence):
  - Client AI mention rate and what it means
  - Primary competitive gap (name the competitor, state the gap)
  - Citation gap count and most significant gap source
  - Sentiment posture
  - Strongest finding (where client leads, if applicable)
- Strategic implication (1-2 sentences): what this means for the client's talent acquisition competitiveness
- Recommendation preview: "This assessment identifies [N] priority actions, led by [top recommendation title]."

**Data source:**
- `scanComparison.clientMentionRate`
- `scanComparison.avgVisibilityScore`
- `scanComparison.avgSentimentScore`
- `scanComparison.entityMentions` (top competitor by mentionRate)
- `scanComparison.citations.gapDomains`
- `recommendations` (count and top item)

**Buyer outcome:** The executive who has 90 seconds reads this page and knows: (1) we have a problem, (2) our competitors are ahead, (3) there are specific things to fix, (4) the rest of this report tells us how. This page alone should justify the assessment cost.

**Language guidance:**
- Write in executive advisory tone, not software-output tone
- Do NOT open with "This report presents..." -- open with the finding
- Do NOT list raw numbers without interpretation ("67% mention rate" means nothing; "mentioned in two-thirds of candidate queries, trailing Acme Corp by 16 points" means everything)
- Frame findings as business risk or opportunity, not technical observations
- Use "assessment" not "audit"

**Example opening (template):**

> [Client Name] has moderate visibility in AI-driven candidate discovery, appearing in [X]% of evaluated queries across [N] candidate-intent scenarios. However, [Top Competitor] holds a [gap]pp advantage in AI mention rate, meaning candidates who ask AI models where to work in [industry] are significantly more likely to hear about [Top Competitor] first.
>
> Key findings:
> - **AI mention rate of [X]%** -- [Client] appears in [count] of [total] candidate queries, indicating [strong/moderate/weak] presence in AI-driven talent discovery
> - **[Gap]pp visibility gap vs. [Competitor]** -- the largest competitive gap identified, concentrated in [theme] queries
> - **[N] citation gaps identified** -- [N] sources actively surface competitors but not [Client], including [top 2 gap domains]
> - **[Positive/Negative/Neutral] sentiment posture** -- AI responses about [Client] carry [sentiment word] tone, which [supports/undermines] candidate interest
>
> This assessment identifies [N] prioritized recommendations. The highest-priority action is [top rec title], which addresses [brief description of why].

### 2. ASSESSMENT SCOPE AND METHODOLOGY (0.5-1 page)

**Content:**
- What was assessed: client company, industry, named competitors (list all)
- Query coverage: total queries evaluated, across which intent themes (name the 6 themes), covering which role families (if role profiles exist)
- Scan coverage: which AI models were queried (from scan metadata), date range of scans
- Scoring methodology (brief):
  - Mention detection: whether the AI response references the client by name
  - Visibility score: how prominently the client appears in the response (0-100)
  - Sentiment score: tone of AI commentary about the client (-1 to +1)
  - Citation extraction: which source URLs the AI cites in its response
- Why this matters: "AI models do not rank employers -- they synthesize answers from hundreds of public signals. This assessment measures how that synthesis represents [Client] relative to competitors."

**Data source:**
- `client.name`, `client.domain`, `client.industry`
- `client.competitors` (names and domains)
- `scanComparison.totalQueries`, `scanComparison.completedQueries`
- `scanRun.model` (AI model used)
- Query cluster data (themes) -- this is not currently passed into the report composer but exists in the database

**Buyer outcome:** Establishes credibility. Enterprise buyers need to trust the methodology before accepting the findings. This section also preempts the "how do you know this?" question that will come up in the executive briefing.

**Language guidance:**
- Keep this concise -- 3-4 short paragraphs, not a technical whitepaper
- Do NOT explain the technology (no mention of APIs, tokens, prompts)
- DO explain the lens: "candidate-intent queries" not "LLM prompts"
- Frame the methodology as the value: "structured assessment across [N] candidate decision scenarios" not "we queried an API"
- The scoring descriptions should be 1 sentence each, plain English

**GAP: Query theme data is not currently passed into `composeReport()`.** The `ReportInput` type needs a new field for query theme breakdown. The data exists -- `QueryCluster` has a name/intent, and queries have themes -- but the report generation action does not query it. This is a **must-have** addition.

**GAP: AI model names are not currently passed into `composeReport()`.** The `ScanRun.model` field exists but is not included in report input. This is a **nice-to-have** -- can be hardcoded or omitted for now.

### 3. VISIBILITY FINDINGS (1-1.5 pages)

**Content:**

**3a. Overall AI Visibility Position**
- Client mention rate with interpretation (not just the number -- what tier it falls in and what that means)
- Average visibility score with interpretation
- Comparison to assessment benchmarks: "Across Antellion assessments, the median employer mention rate is [X]%. [Client] falls [above/below/at] this benchmark." (Note: this requires aggregate data across clients, which does not exist yet. For now, use tier-based framing.)

**3b. Visibility by Query Theme (new)**
- Table or breakdown showing mention rate per query theme:
  - Reputation and Brand: X%
  - Compensation and Benefits: X%
  - Hiring Process: X%
  - Role Expectations: X%
  - Culture: X%
  - Competitor Comparison: X%
- Identify the theme where the client is strongest and weakest
- Narrative: "AI models are most likely to mention [Client] when candidates ask about [strongest theme], but [Client] is nearly absent from [weakest theme] queries."

**3c. Sentiment Analysis**
- Average sentiment score with interpretation
- What positive/negative sentiment means for candidate decision-making
- If negative: "Candidates who encounter [Client] in AI responses receive a [negative/mixed] impression, which may reduce application intent even when the company is mentioned."

**Data source:**
- `scanComparison.clientMentionRate`
- `scanComparison.avgVisibilityScore`
- `scanComparison.avgSentimentScore`
- Per-theme mention rate (NEW -- requires computation from scan results joined with query cluster themes)

**Buyer outcome:** The buyer sees exactly where they stand. Not just "your number is X" but "your number is X, which means Y for your hiring." This is the section that creates the "we have a problem" moment.

**Language guidance:**
- Lead with the business implication, then support with the data
- "Low" visibility is "candidates asking AI about [industry] employers are unlikely to hear about [Client]" -- not "mention rate is 23%"
- Do NOT present visibility scores in isolation. Always contextualize: is this good? bad? relative to what?
- Use "candidate queries" not "evaluated prompts"
- The per-theme breakdown should explicitly name the 6 themes in plain English, not enum values

**GAP: Per-theme mention rate is not computed.** The scan comparison currently aggregates all results. To break by theme, the report generation action needs to join scan results back to their queries and query clusters, then compute mention rates per theme. The data path exists: `ScanResult.queryId -> Query.queryClusterId -> QueryCluster.name/intent`. This is a **must-have** for the Assessment deliverable.

### 4. COMPETITIVE ANALYSIS (1.5-2 pages)

**Content:**

**4a. Competitive Visibility Comparison**
- Table comparing all entities (client + each competitor):

| Company | AI Mention Rate | Visibility Score | Sentiment | Gap vs. Client |
|---|---|---|---|---|
| Client | X% | X/100 | Positive | -- |
| Competitor A | X% | X/100 | Neutral | +Xpp |
| Competitor B | X% | X/100 | Negative | -Xpp |

- Narrative identifying the primary competitive threat: which competitor has the largest advantage and in what areas
- Where the client has an advantage: explicitly call out any competitors the client leads

**4b. Per-Competitor Profiles (Competitor Visibility Scorecards -- Bonus 1)**
For each competitor, a brief profile (3-5 sentences):
- Mention rate and how it compares to client
- Sentiment posture
- Notable citation sources that surface this competitor but not the client
- What this competitor's visibility advantage appears to be (which themes they dominate, if theme-level data is available)

**Data source:**
- `scanComparison.entityMentions` (all entries, including per-entity mentionRate and mentionCount)
- Per-competitor sentiment and visibility scores (NEW -- requires per-entity score computation)
- `scanComparison.citations` (to identify which gap domains are associated with which competitor)

**Buyer outcome:** The buyer can walk into a meeting and say "Competitor X has a 73% AI mention rate vs. our 31%. Here is why." The per-competitor profiles are the "Competitor Visibility Scorecards" bonus from the playbook -- delivering them inside the main report rather than as a separate document is stronger.

**Language guidance:**
- The comparison table should be the visual anchor of this section
- Name competitors by name, always. Never say "Competitor A" in a paid assessment -- the buyer is paying for specificity
- Frame gaps as competitive displacement risk: "Candidates who ask AI about [role] in [industry] are [X]x more likely to hear about [Competitor] than [Client]"
- Do NOT present this as a leaderboard. Frame it as a competitive landscape with implications.

**GAP: Per-competitor visibility and sentiment scores are not computed.** The current `EntityMentionStats` type only has `name`, `isClient`, `mentionCount`, and `mentionRate`. It does not carry `avgVisibilityScore` or `avgSentimentScore` per entity. To compute these, we would need the scan results to carry competitor-level scores (currently the visibility and sentiment scores are per-response, not per-entity within a response). The mention data comes from `metadata.competitorMentions` which only has `mentioned: boolean`. **For launch, the comparison table should show mention rate and gap -- visibility and sentiment scores should be shown only for the client.** Per-competitor sentiment is a **nice-to-have** for V2.

**GAP: Citation gap attribution to specific competitors is not computed.** Gap domains are currently identified as "domains cited when client is NOT mentioned" -- they are not attributed to which competitor's mention they appeared alongside. This attribution would require the comparison computation to track which competitor was mentioned in the same response where the gap domain appeared. This is a **nice-to-have** -- for launch, gap domains can be listed without competitor attribution.

### 5. CITATION ANALYSIS (1-1.5 pages)

**Content:**

**5a. Citation Landscape**
- Total unique domains cited across all AI responses
- Top cited domains by frequency (table, top 10):

| Source | Times Cited | Cites Client | Cites Competitors | Type |
|---|---|---|---|---|
| glassdoor.com | 12 | Yes | Yes | Review site |
| linkedin.com | 9 | Yes | Yes | Professional network |
| levels.fyi | 6 | No | Yes | Compensation data |

**5b. Citation Gaps (Citation Source Playbook -- Bonus 2)**
- Gap domains listed with:
  - Domain name
  - Source type (review site, career blog, compensation platform, social profile, press/media, technical community)
  - Frequency (how often this source appears in AI responses)
  - Recommended action (what the client should do to gain presence on this source)
- Framing: "These sources are actively informing AI responses about your competitors but not about [Client]. Gaining presence on these sources is the most direct path to improving AI visibility."

**5c. Client-Exclusive Sources**
- Sources that only cite the client (competitive advantage to protect)
- Brief note on why these matter: "These sources currently give [Client] an advantage. Ensure continued presence."

**5d. Shared Sources**
- Sources that cite both client and competitors
- Note: "These sources influence AI responses for all assessed companies. Quality and recency of content on these platforms affects relative positioning."

**Data source:**
- `scanComparison.citations.domainFrequency` (top domains by count)
- `scanComparison.citations.gapDomains`
- `scanComparison.citations.clientExclusiveDomains`
- `scanComparison.citations.sharedDomains`
- Source type classification (NEW -- requires either manual mapping or heuristic classification of domain types)

**Buyer outcome:** The buyer gets an actionable target list. Instead of "improve your employer brand" (useless), they get "gain presence on levels.fyi, Blind, and Built In because those sources are informing AI about your competitors right now." This is the Citation Source Playbook bonus from the playbook, delivered as a section of the main report.

**Language guidance:**
- Lead with gaps -- they are the most actionable finding
- Make the recommended actions specific: "Create or claim your employer profile on [domain]" or "Publish engineering content that would be indexed by [domain]"
- Do NOT just list domains. Classify them and explain why they matter.
- Use "sources" not "domains" in buyer-facing language
- The table format is critical -- this section is useless as a paragraph of domain names (which is what the current report produces)

**GAP: Source type classification does not exist.** The `CitationSource` model has a `sourceType` field, but it is nullable and likely unpopulated for most citations. A heuristic classifier (mapping known domains to types: glassdoor.com -> "Review site", medium.com -> "Blog platform", etc.) would solve this. **Must-have for launch** -- even a basic 20-domain mapping table would dramatically improve this section.

**GAP: Recommended actions per gap domain do not exist.** Currently gap domains are just listed. For the Assessment, each gap domain should have a brief action recommendation. This could be template-based by source type (e.g., all "Review site" gaps get "Create or update your employer profile"). **Must-have for launch** -- can be implemented as a static mapping from source type to recommended action template.

- Narrative: "Candidates evaluating [Client] through AI receive the strongest signal about [strongest theme], but are least likely to encounter [Client] when asking about [weakest theme]. This gap indicates [interpretation]."

**Data source:**
- Per-theme mention rates (NEW -- same computation needed for Section 3b)
- Query cluster data (theme names, query counts)

**Buyer outcome:** The buyer sees which candidate decision moments they are winning and losing. A VP of TA can look at this and immediately say "we need to address our culture narrative in AI" -- which is a specific, actionable insight they cannot get anywhere else.

### 7. PRIORITIZED RECOMMENDATIONS (1-2 pages)

**Content:**

**7a. Priority Actions**
- Top 3 recommendations presented in detail:
  - Title (action-oriented: "Close citation gaps on compensation platforms" not "CONTENT_GAP: Close citation gaps")
  - Why this matters (business impact, 2-3 sentences)
  - What to do (specific actions, 2-4 bullets)
  - Expected impact (what changes if this is done)
  - Effort level (Low / Medium / High, with brief justification)
  - Priority (Critical / High / Medium / Low)

**7b. Additional Recommendations**
- Remaining recommendations in a summary table (same format as current export, but with cleaner column labels)

**7c. Recommended Next Steps**
- Brief closing section (3-4 sentences):
  - "The findings in this assessment represent a point-in-time snapshot. AI models continuously update based on new data."
  - "Quarterly re-assessment allows [Client] to measure progress against these recommendations and identify new competitive developments."
  - "[Client] should prioritize the top [N] recommendations in the next 90 days and schedule a follow-up assessment to measure impact."

**Data source:**
- `recommendations` (all fields: category, priority, title, description, impact, effort)
- Current recommendation generation logic in `generateRecommendations()` is solid -- the data is good, the presentation needs to improve

**Buyer outcome:** The buyer leaves with a clear action plan. The "recommended next steps" section is the natural bridge to the upsell (quarterly monitoring / Intelligence tier). The top-3 detail format ensures the buyer's team knows exactly what to do first.

**Language guidance:**
- Lead with "what to do" not "what we found"
- Recommendation titles should be imperative: "Establish presence on compensation data platforms" not "Content gap identified"
- The "why this matters" should reference the specific finding from earlier in the report: "As identified in Section 4, [Competitor] has a [X]pp advantage in AI mention rate, driven in part by their presence on [gap domains]."
- Do NOT use enum values in buyer-facing text. "CONTENT_GAP" should render as "Content and Citation Gap" or similar
- The effort field should include a time horizon: "Medium -- 4-6 weeks" not just "Medium"
- "Recommended Next Steps" should feel advisory, not salesy. It should read like a consultant's closing recommendation, not a subscription upsell.

**IMPROVEMENT: The current recommendation generator should be enhanced to produce richer descriptions.** Currently each recommendation has a 1-2 sentence description. For the top 3, the report needs 2-3 sentence "why this matters" + 2-4 bullet "what to do" items. This could be achieved by adding a `details` field to `GeneratedRecommendation` or by having the report composer generate extended content for the top N recommendations. **Must-have for launch.**

## 4. IMPLEMENTATION GUIDANCE

### Report Data Structure Changes

The `ReportSection` type needs to evolve from plain text to structured data:

```typescript
// Current
interface ReportSection {
  heading: string;
  body: string;
}

// Target
interface ReportSection {
  heading: string;
  body: string;
  subsections?: ReportSubsection[];
}

interface ReportSubsection {
  heading: string;
  body?: string;
  table?: ReportTable;
  items?: string[];  // bullet list
}

interface ReportTable {
  headers: string[];
  rows: (string | number | null)[][];
}
```

This allows sections to contain tables (competitor comparison, citation landscape, query intent map) alongside narrative text.

### ReportInput Changes

```typescript
interface ReportInput {
  clientName: string;
  clientDomain: string;
  industry?: string;
  scanComparison: ScanComparisonResult;

  // NEW fields:
  queryThemeBreakdown: QueryThemeStats[];     // per-theme mention rates
```

### Source Type Classifier

A minimal heuristic classifier for citation domains:

```typescript
const KNOWN_SOURCE_TYPES: Record<string, string> = {
  "glassdoor.com": "Employee review site",
  "indeed.com": "Job board and review site",
  "linkedin.com": "Professional network",
  "blind.com": "Anonymous employee forum",
  "levels.fyi": "Compensation data platform",
  "builtin.com": "Tech employer directory",
  "comparably.com": "Employee review and culture site",
  "payscale.com": "Compensation research",
  "medium.com": "Blog platform",
  "github.com": "Technical community",
  "stackoverflow.com": "Developer community",
  "reddit.com": "Community forum",
  "techcrunch.com": "Technology press",
  "forbes.com": "Business press",
  "bloomberg.com": "Business and financial press",
  "crunchbase.com": "Company intelligence",
  "wellfound.com": "Startup job platform",
  "teamblind.com": "Anonymous employee forum",
  "vault.com": "Career intelligence",
  "fairygodboss.com": "Women's career community",
};

// Fallback pattern matching
function classifyDomain(domain: string): string {
  if (KNOWN_SOURCE_TYPES[domain]) return KNOWN_SOURCE_TYPES[domain];
  if (domain.includes("career") || domain.includes("jobs")) return "Career site";
  if (domain.includes("review")) return "Review site";
  if (domain.includes("news") || domain.includes("press")) return "Press/media";
  if (domain.includes("blog")) return "Blog";
  return "Other source";
}
```

## 5. LANGUAGE STYLE GUIDE

### Executive Summary

**DO:**
- "trails [Competitor] by 24 percentage points in AI visibility"
- "candidates researching [industry] employers through AI are 2.4x more likely to hear about [Competitor]"
- "7 citation sources actively inform AI about competitors but not [Client]"
- "[Client] has a meaningful opportunity to close competitive gaps through targeted content and presence improvements"

**DO NOT:**
- "mention rate is 0.42"
- "average visibility score of 55 out of 100"
- "the client was mentioned in 12 of 28 scan results"
- "based on our analysis of LLM outputs"
- "the data shows"

### Findings Sections

**DO:**
- Write in advisory tone: "This gap represents a candidate decision risk"
- Contextualize every number: "29% mention rate in compensation queries -- meaning fewer than 1 in 3 candidates who ask AI about pay in [industry] will hear about [Client]"
- Name competitors by name, always
- Use "candidate queries" not "prompts" or "scan results"
- Use "AI responses" not "LLM outputs"
- Use "sources" not "domains" or "URLs"

**DO NOT:**
- Use "scan" as a buyer-facing term. Use "assessment" or "evaluation"
- Use "metadata" or any internal system terms
- Present lists without interpretation
- Use passive voice when active is clearer: "Candidates encounter [Client]..." not "The client was encountered by..."
- Use hedge language: "likely" and "may" are fine, but avoid "we think" or "it seems"

### Recommendations

**DO:**
- Lead with imperative verbs: "Close," "Establish," "Strengthen," "Address"
- Connect each recommendation to a specific finding: "As identified in the competitive analysis..."
- Include effort time horizons: "Medium effort -- 4-6 weeks, primarily content team"
- Make actions specific: "Create employer profiles on levels.fyi, Blind, and Built In" not "Improve your online presence"

**DO NOT:**
- Use category enum values in text: "CONTENT_GAP" should never appear in the report
- Use generic advice: "Improve your employer brand" is never acceptable
- Present recommendations without connecting them to findings
- Use priority labels without explanation: say "Critical priority -- this is the largest competitive gap" not just "CRITICAL"

### Tone Summary

The report should read as if it were written by a senior strategy consultant who specializes in employer brand and competitive intelligence. It should be:

- **Specific:** every claim backed by a data point from the assessment
- **Interpretive:** every data point connected to a business implication
- **Actionable:** every finding connected to a recommendation
- **Confident:** advisory tone, not hedging or qualifying excessively
- **Concise:** executive-length paragraphs (3-5 sentences max per block)

The report should NOT read as if it were generated by software. The current report reads like a software output summary. The target is a strategic advisory document.

## 8. WHAT THE REPORT MUST NOT BECOME

To protect the quality and positioning of the Assessment deliverable, the following anti-patterns must be actively prevented:

1. **Not an SEO report.** Do not use language like "rankings," "SERP," "keywords," "backlinks," or "domain authority." AI visibility is a different surface.

2. **Not a data dump.** Every table should have narrative interpretation. Every metric should have business context. If a number appears, the next sentence explains what it means.

3. **Not a generic template.** Every paragraph should reference the client by name, reference specific competitors, and reference specific data points. If you could swap in any company name and the text still works, it is too generic.

4. **Not a software report.** No database IDs, no enum values, no technical metadata, no "system-generated" language. The report should feel handcrafted even when it is composed algorithmically.

5. **Not a sales pitch.** The "Recommended Next Steps" section should feel like strategic advice, not a product upsell. The quality of the report itself is the upsell.
